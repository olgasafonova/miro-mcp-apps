## Decisions worth preserving

- **Polyglot stance over language-purity for MCP Apps.** Go SDK v1.6.1 has zero MCP Apps capability helpers (verified via `gh api search/code` against `modelcontextprotocol/go-sdk`); official ext-apps SDK and four Anthropic-shipped Agent Skills (`create-mcp-app`, `add-app-to-server`, `migrate-oai-app`, `convert-web-app`) are TS-only. The wire format IS language-agnostic, but implementing manually in Go throws away the official tooling. Go `miro-mcp-server` stays as the 89-CRUD-tools production surface; TS `miro-mcp-apps` is the visual-pattern demo. Both reuse `MIRO_ACCESS_TOKEN`. No coupling.
- **TS path ~1h vs Go path 5-7d.** This was the structural payoff of the language choice — the official `basic-server-vanillajs` example + ext-apps SDK + Vite did most of the work.
- **Vanilla JS over React for the UIs.** No build chain ceremony, ~340KB unminified per HTML bundle (~83KB gzipped), enough for card + bar chart + table. Reach for React only if a future tool needs complex interactivity (the connectors graph might).
- **`tsc` substituted for `bun` in the example's build script.** Local pilot, not npm-publishable; `tsc -p tsconfig.server.json` emits JS to `dist/`, `node dist/main.js` runs it. If publish-to-npm becomes a goal later, swap back to `bun build` (or `esbuild`) for the server bundling step.
- **Token-in-config-env, not shell-env-passthrough.** Claude Desktop config convention (seen in other entries: BOLAGSVERKET_*, N8N_API_KEY, AZURE_DEVOPS_PAT) is embedded credentials. New `miro-apps` entry follows this. Token written via python script that pulled from `$MIRO_ACCESS_TOKEN` — never echoed to chat.
- **Miro accent color `#FFD02F`** in `src/global.css` `--color-accent`. Brand yellow; `--color-text-on-accent: #050038` (Miro dark navy) for contrast. Reuse for new tools.

## Watch-outs

- **Don't auto-restart Claude Desktop without asking.** It's a UI session you may have stuff open in.
- **The `MIRO_ACCESS_TOKEN` is an OAuth token (JWT shape, 58 chars).** Shorter lifespan than a PAT. First 401 = rotation time. The Go server has the same dependency; if Go works, TS works.
- **`vite-plugin-singlefile` bundles per `INPUT=*.html`** — `build:ui` runs vite twice (one per HTML), `emptyOutDir: false` keeps the previous output. Don't add `emptyOutDir: true` or each bundle wipes the previous one.
- **Server.ts imports use `.js` extension** (`./miro-client.js` not `./miro-client.ts`) per NodeNext convention. `tsx` resolves `.js` → `.ts` in dev; compiled output uses the real `.js`. Don't "fix" the imports to `.ts` — it'll break runtime.
- **HTTP mode binds `0.0.0.0` without `allowedHosts`** — SDK prints a warning at startup. Fine for local pilot. Add `allowedHosts` config before any non-local deployment.
- **No automated test suite yet.** New tool additions are validated only by `npm run build` (typecheck + bundle). Eyes-on testing required per host.
- **Anti-slop check for next session's tool names + descriptions:** the existing two tool descriptions use the "USE WHEN" pattern from `gleif-mcp-server/tools/definitions.go`. Keep the same shape for the 4 new tools so Claude's tool selector has consistent guidance.
